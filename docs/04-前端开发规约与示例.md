# 04-前端开发规约与示例 (Vue 3)

本文档适用于 `frontend` 模块，重点约束页面拆分、状态管理、接口调用、交互一致性和移动端适配。

## 1. 模块职责

- 承担上传、讲课、问答、学习进度等交互页面。
- 展示后端权威状态，**不自行维护一套独立业务状态机**。
- 统一管理请求、错误提示、加载态和用户反馈。

## 2. 当前目录约定与演进

```text
frontend/src/
├── assets
├── components          # 复用组件
├── constans            # 建议后续统一迁移为 constants
├── router
├── stores              # Pinia 状态管理
├── utils               # 无状态工具函数
└── views               # 页面级组件
```

## 3. 状态管理规范

### 3.1 Store 适用范围

- 用户信息、当前课件信息、当前讲课会话、全局主题。

### 3.2 不应放进 Store 的内容

- 临时输入框草稿、局部弹窗开关、单页内一次性状态。

**代码示例：Pinia Store 同步后端权威状态**

```typescript
// stores/lecture.ts
export const useLectureStore = defineStore('lecture', {
  state: () => ({
    sessionId: '',
    status: 'IDLE' as 'IDLE' | 'PLAYING' | 'INTERRUPTED',
  }),
  actions: {
    // 从后端拉取最新状态，覆盖本地
    async syncSessionStatus() {
      const data = await getSessionDetail(this.sessionId);
      this.status = data.status; // 完全以后端为准
    }
  }
});
```

## 4. 接口调用规范

### 4.1 请求入口

- 所有 HTTP 请求统一走 `utils/request.js` 或专门的 api 模块。
- 不在页面组件内手写基础地址。

### 4.2 响应处理

- 统一处理 `code` / `message` / `data`。
- 非 0 响应统一进入错误提示流程。

**代码示例：Axios 拦截器统一处理**

```typescript
// utils/request.ts
request.interceptors.response.use(
  (response) => {
    const res = response.data;
    if (res.code !== 0) {
      ElMessage.error(res.message || '请求失败');
      return Promise.reject(new Error(res.message));
    }
    return res.data;
  }
);
```

## 5. 交互规范与状态展示

### 5.1 状态展示

- 使用后端返回的状态枚举，不自行推断流程完成。
- 前端显示文案与状态值建立映射表，集中维护。

**代码示例：状态映射表**

```typescript
// utils/constants.ts
export const LECTURE_STATUS_MAP = {
  IDLE: { text: '待开始', color: 'info' },
  PLAYING: { text: '讲解中', color: 'success' },
  INTERRUPTED: { text: '倾听中', color: 'warning' }
};
```

### 5.2 异步流程

- 所有异步动作必须有加载态。
- 上传、解析、讲课、问答必须有明确状态提示。

## 6. 移动端与嵌入式场景

- 重点兼容移动端浏览器和 WebView。
- 检查输入法遮挡、音视频控制区点击范围。
- 交互元素最小点击区建议不小于 44x44。

## 7. 不该做什么

- 不在多个页面重复写同样的请求逻辑。
- 不把页面局部状态全部塞进全局 Store。
- 不在前端独立维护一套后端状态机。
- 不直接把后端原始异常文本展示给用户。
