## 1. API 规范

### 1.1 通用约定
- **RESTful 风格**：资源命名使用名词复数，路径全部小写，单词间用短横线分隔（如 `/api/v1/courseware`）。
- **版本控制**：所有 API 前缀 `/api/v1/`，后续迭代升级时增加 `/api/v2/`。
- **请求方式**：
  - `GET`：查询资源
  - `POST`：创建资源或执行动作
  - `PUT`：全量更新
  - `PATCH`：部分更新
  - `DELETE`：删除资源
- **字符编码**：统一 UTF-8。
- **Content-Type**：请求与响应均使用 `application/json`（文件上传使用 `multipart/form-data`）。

### 1.2 请求与响应格式

#### 成功响应
```json
{
  "code": 0,
  "message": "success",
  "data": { ... }   // 返回的数据，可为 null 或对象/数组
}
```

#### 错误响应
```json
{
  "code": 40001,
  "message": "参数错误：课件ID不能为空",
  "data": null
}
```
- 错误码定义：
  - `0`：成功
  - `400xx`：客户端错误（参数、业务校验）
  - `500xx`：服务端错误
  - `401xx`：认证授权错误

#### 分页响应（列表类接口）
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [ ... ],
    "total": 100,
    "page": 1,
    "pageSize": 20
  }
}
```

### 1.3 核心接口示例（部分）

#### 上传课件
- **URL**：`POST /api/v1/courseware/upload`
- **请求**：`multipart/form-data`
  - `file`: 课件文件（PPTX/PDF）
  - `name`: 可选，课件名称
- **响应**：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "coursewareId": "cware_123456",
    "status": "UPLOADED"
  }
}
```

#### 开始讲课
- **URL**：`POST /api/v1/lecture/start`
- **请求**：
```json
{
  "coursewareId": "cware_123456",
  "userId": "user_001"
}
```
- **响应**：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "sessionId": "sess_789",
    "currentNode": {
      "nodeId": "node_001",
      "pageIndex": 1,
      "content": "讲解内容文本",
      "audioUrl": "https://.../tts.mp3"  // 可选
    }
  }
}
```

#### 文本问答
- **URL**：`POST /api/v1/qa/ask-text`
- **请求**：
```json
{
  "sessionId": "sess_789",
  "question": "什么是递归？"
}
```
- **响应**：
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "answer": "递归是指函数直接或间接调用自身的一种编程技巧...",
    "evidence": [
      {
        "source": "page_3",
        "text": "递归是一种常见的算法思想..."
      }
    ],
    "latencyMs": 1250
  }
}
```

#### 恢复讲课
- **URL**：`POST /api/v1/lecture/resume`
- **请求**：
```json
{
  "sessionId": "sess_789",
  "resumeToken": "node_001|offset_123"   // 可选，默认从上次中断处恢复
}
```
- **响应**：同开始讲课的响应

### 1.4 Java 与 Python 服务间通信
- **同步调用**：Java 通过 HTTP 调用 Python 服务（REST），统一使用 Feign 或 RestTemplate。
- **接口约定**：Python 也遵循同样的 API 格式（code/message/data），路径独立（如 `/python/v1/parse`）。
- **错误处理**：Python 返回非 0 时，Java 应记录日志并向上层返回友好错误。

## 2. Java 后端规范

### 2.1 项目结构（按模块）
```
src/main/java/com/yourcompany/
├── controller/       # 控制器，处理 HTTP 请求
├── service/          # 业务逻辑层
├── repository/       # 数据访问层（JPA/MyBatis）
├── dto/              # 数据传输对象（请求/响应）
├── entity/           # 数据库实体
├── config/           # 配置类
├── util/             # 工具类
└── exception/        # 自定义异常及全局处理
```

### 2.2 分层职责
- **Controller**：仅接收请求参数、调用 Service、返回响应。不写业务逻辑。
- **Service**：实现业务逻辑，事务管理，调用 Repository 和 Python 客户端。
- **Repository**：数据库操作，使用 Spring Data JPA 或 MyBatis Plus。

### 2.3 异常处理
- 统一使用 `@RestControllerAdvice` 捕获异常，转换为统一错误响应。
- 自定义业务异常：`BusinessException`（code + message）。
- 对于 Python 服务调用异常，包装为 `ServiceException`。

### 2.4 日志
- 使用 SLF4J + Logback。
- 日志级别：开发环境 DEBUG，生产 INFO。
- 关键操作（上传、解析、问答）必须记录入参与出参。

### 2.5 与 Python 服务的 HTTP 客户端
- 定义 `PythonApiClient` 接口（Feign 或 RestTemplate）。
- 统一设置超时时间（连接超时 5s，读取超时 30s）。
- 增加重试机制（最多 2 次）。

## 3. Python 后端规范

### 3.1 项目结构
```
python-service/
├── app/
│   ├── main.py              # FastAPI 入口
│   ├── api/                 # 路由定义
│   │   └── v1/
│   │       ├── parse.py
│   │       ├── qa.py
│   │       └── lecture.py
│   ├── services/            # 业务逻辑（解析、RAG等）
│   │   ├── parser.py
│   │   ├── rag.py
│   │   └── tts.py
│   ├── models/              # 数据模型（Pydantic）
│   ├── utils/               # 工具
│   └── config.py            # 配置
├── requirements.txt
└── Dockerfile
```

### 3.2 接口定义
- 使用 FastAPI，自动生成 Swagger 文档（`/docs`）。
- 路径前缀：`/python/v1`。
- 请求/响应模型使用 Pydantic，保持与 Java 侧格式一致（code/message/data）。

### 3.3 错误处理
- 定义统一响应模型 `BaseResponse`。
- 异常时使用 `HTTPException`，并返回符合约定的 JSON。

### 3.4 日志
- 使用 `logging` 模块，输出到控制台及文件。
- 关键接口记录请求耗时。

## 4. 数据库规范

### 4.1 命名
- 表名：小写，下划线分隔（如 `courseware`, `qa_record`）。
- 字段名：小写，下划线分隔（如 `courseware_id`, `created_at`）。
- 主键：统一使用 `id`（UUID 或自增长整型）。
- 时间字段：`created_at`, `updated_at`（datetime 类型）。

### 4.2 索引
- 外键字段、查询频繁字段建立索引。
- 唯一约束：`courseware_id` 等。

### 4.3 示例表结构（DDL）
```sql
CREATE TABLE courseware (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    courseware_id VARCHAR(64) NOT NULL UNIQUE,
    name VARCHAR(255),
    file_path VARCHAR(512),
    status VARCHAR(20) DEFAULT 'UPLOADED',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status (status)
);

CREATE TABLE qa_record (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(64) NOT NULL,
    question TEXT NOT NULL,
    answer TEXT,
    evidence TEXT,   -- JSON 格式存储
    latency_ms INT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_session (session_id)
);
```

## 5. 前端规范

### 5.1 技术栈
- Vue 3 + Vite 或 React + Vite。
- UI 组件库：可选 Vant（移动端）或自定义。
- 状态管理：Pinia（Vue）或 Zustand（React）。
- HTTP 请求：Axios，统一拦截器处理 token、错误。

### 5.2 目录结构
```
src/
├── api/           # 接口定义
├── assets/        # 静态资源
├── components/    # 公共组件
├── views/         # 页面视图
├── router/        # 路由
├── stores/        # 状态管理
├── utils/         # 工具函数
└── styles/        # 全局样式
```

### 5.3 移动端适配
- 使用 `viewport` 设置 `width=device-width, initial-scale=1.0`。
- 使用 `rem` 或 `vw/vh` 进行响应式布局。
- 交互元素最小点击区域 44×44 pt。

### 5.4 API 调用规范
- 所有 API 调用集中在 `api/` 目录下，按模块拆分。
- 请求拦截器自动添加认证头（如需要）。
- 响应拦截器统一处理 `code`，非 0 时弹出错误提示。

## 6. 协作规范

### 6.1 Git 分支管理
- `main`：生产环境代码，仅接受 merge request。
- `develop`：开发主分支，日常集成。
- `feature/*`：功能分支（如 `feature/upload-parse`），从 `develop` 拉出，完成后合并回 `develop`。
- `hotfix/*`：紧急修复分支。

### 6.2 提交信息格式
```
<type>(<scope>): <subject>

type: feat | fix | docs | style | refactor | test | chore
scope: 模块名（如 api, frontend, db）
subject: 简短描述
```


#### `<type>` 类型
描述本次提交的性质，必须使用以下词汇之一：

| type | 含义 | 示例场景 |
|------|------|----------|
| `feat` | 新增功能 | 实现课件上传接口 |
| `fix` | 修复 Bug | 修复问答接口超时问题 |
| `docs` | 文档相关 | 更新 API 文档 |
| `style` | 代码格式调整（不影响逻辑） | 缩进、空格、分号 |
| `refactor` | 代码重构（不改变功能） | 抽取公共方法、优化结构 |
| `test` | 增加或修改测试 | 添加单元测试、压测脚本 |
| `chore` | 构建、工具、依赖等杂项 | 修改 Dockerfile、升级依赖 |

#### `<scope>` 范围
标识本次提交影响的模块，用单词或简写，便于快速定位。常用范围（根据你的项目）：
- `api`：接口定义
- `backend`：Java 后端
- `parser`：Python 解析模块
- `rag`：Python 问答模块
- `frontend`：前端
- `db`：数据库相关
- `config`：配置
- `deploy`：部署脚本

#### `<subject>` 简短描述
- 用中文或英文（团队统一即可），建议用中文更直观
- 不超过 50 个字符
- 使用祈使句，不加句号
- 描述“做了什么”，而非“做了什么”的细节

#### 示例

```bash
# 新增功能
feat(backend): 实现课件上传接口
feat(parser): 增加PPT页面分解功能

# 修复问题
fix(rag): 修复问答检索时未过滤当前讲稿的问题
fix(frontend): 修复移动端问答框键盘遮挡问题

# 文档
docs(api): 补充文本问答接口的响应示例

# 重构
refactor(backend): 抽取Python调用客户端为独立类

# 配置/工具
chore(deploy): 添加Docker Compose环境配置
```


### 6.3 接口文档维护
- 使用 Swagger/OpenAPI（Java 用 Springdoc，Python 用 FastAPI 自带）。
- 在前端项目中，同步维护一份 Markdown 格式的接口文档，方便查阅。
