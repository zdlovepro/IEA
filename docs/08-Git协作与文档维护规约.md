# 08-Git协作与文档维护规约

## 1. 分支规范

- `main`：稳定可交付版本，只接受经过充分测试的合并。
- `develop`：日常集成分支，所有 feature / fix 分支基于此。
- `feature/*`：功能开发分支，例如 `feature/ppt-upload`。
- `fix/*`：问题修复分支，例如 `fix/session-status-sync`。
- `hotfix/*`：紧急修复分支，直接基于 `main`，修复后同步合并回 `develop`。

## 2. 提交规范

**格式**：`<type>(<scope>): <subject>`

**Type 建议**：
- `feat`：新功能
- `fix`：Bug 修复
- `docs`：文档变更
- `refactor`：重构（不改变外部行为）
- `test`：测试相关
- `chore`：构建/工具配置
- `style`：代码格式调整（不影响逻辑）

**示例**：
```text
feat(backend): 新增课件解析任务创建接口
fix(frontend): 修复移动端问答输入框遮挡问题
docs(project): 补充跨服务开发规约
refactor(python-service): 拆分向量检索与答案生成逻辑
```

## 3. 合并要求

- 合并前至少完成自测，确保主链路可用。
- 不允许带明显调试代码、无用日志、临时配置进入主分支。
- 涉及接口变更时，前后端需确认字段兼容，并同步更新文档。
- 大改动优先小步提交，避免一次提交跨多个模块且缺少说明。
- 合并到 `main` 前必须通过 CI 测试（单元测试 + 集成测试）。

## 4. Code Review 关注点

审查者至少检查以下方面：
- 是否违反分层职责（如 Controller 写业务逻辑、Service 直接操作 HttpServletRequest）。
- 是否引入硬编码和隐式状态（魔法数字、无注释的状态值）。
- 是否缺少错误处理和日志（特别是外部调用、状态转换失败时）。
- 是否改动了接口契约但未同步文档（路径、字段、枚举、错误码）。
- 是否存在明显的并发、重复提交、状态错乱风险（如缺少幂等处理）。
- 是否包含敏感信息（密钥、Token 硬编码，日志打印密码）。

## 5. 文档维护规约

### 5.1 必须同步更新文档的场景
- 新增或修改对外接口（路径、请求/响应结构、错误码）。
- 增加新的状态枚举或修改现有状态含义。
- 引入新第三方依赖或新中间件（如 Redis、消息队列）。
- 调整部署方式、环境变量、配置文件结构。
- 修改核心业务流程（如课件解析、讲课中断恢复的步骤变化）。

### 5.2 文档结构与要求
- `README.md`：项目介绍、快速启动、模块说明。
- `docs/api.md`：所有对外接口的详细说明（请求示例、响应示例、错误码）。
- `docs/SRS.md`：需求与验收目标（可选，根据项目规模）。
- `docs/规约.md`：研发执行规范（即本文档所在系列）。
- 设计图、时序图、状态图建议补充到 `docs/design/` 目录。

### 5.3 同步时效
- 接口合并到 `develop` 前，API 文档必须更新完毕。
- 状态枚举变更时，需在 `docs/api.md` 和代码注释中同步说明。
- 如果实际代码与文档不一致，以修正文档和代码其中之一的方式尽快收敛，不允许长期漂移。


## 6. 各层“不该做什么”（快速检查清单）

### 6.1 Controller（Java）不该做什么
- 不直接操作数据库（无 Mapper 调用）。
- 不拼复杂业务对象（交给 Service）。
- 不吞异常（交给全局异常处理器）。
- 不返回数据库 Entity 给前端。

### 6.2 Service（Java）不该做什么
- 不写页面文案、不依赖具体前端展示细节。
- 不把数据库字段名直接当成所有场景的返回契约。
- 不在事务中发起长耗时外部调用（应异步化）。

### 6.3 Python 算法服务不该做什么
- 不承担平台账户权限管理、页面状态编排。
- 不把模型原始输出（未经校验、未经兜底）直接返回。
- 不在一个函数内堆满解析、检索、生成、存储全部逻辑。

### 6.4 前端页面不该做什么
- 不重复维护后端状态机（以后端返回为准）。
- 不散落请求地址和业务常量（统一放到 `api/` 和 `constants/`）。
- 不直接处理与页面无关的全局副作用（交给 Store 或工具函数）。

## 7. 违规处理与规约演进

- 代码合并前若发现明显违反本规约，Reviewer 有权打回修改。
- 若团队达成共识需调整规约，应通过 PR 修改本文档并说明原因，全员知悉后生效。
- 本规约自即日起作为团队默认研发规范执行，后续随项目演进持续完善。
