
# 02-后端开发规约与示例 (Java)

本文档适用于 `backend` 模块，重点约束 Java 后端在接口层、业务层、数据层、日志、异常、配置和联调中的实现方式。

## 1. 模块职责

- 对外暴露统一 API。
- 负责业务编排、权限控制、状态管理、数据持久化。
- 调用 Python 服务和第三方服务。
- 为前端提供权威状态与结果，不把流程判断下放到前端。

## 3. 分层职责

### 3.1 Controller

接收请求、校验参数、调用 Service、返回统一响应。

- 不直接操作数据库。
- 不写核心业务逻辑。
- 不返回 Entity。

**代码示例：Controller 编写规范**

```java
@RestController
@RequestMapping("/api/v1/lecture")
@Slf4j
public class LectureController {
    @Autowired
    private LectureService lectureService;

    @PostMapping("/start")
    public Result<LectureSessionVO> startLecture(@Valid @RequestBody StartLectureRequest request) {
        log.info("开始讲课请求，课件ID: {}, 用户ID: {}", request.getCoursewareId(), request.getUserId());

        // 直接调用 Service，Controller 保持简短
        LectureSessionVO sessionVO = lectureService.startLectureSession(request);

        return Result.success(sessionVO);
    }
}
```

### 3.2 Service

承载业务规则和流程编排。

- 校验状态流转合法性。
- 控制事务边界。

### 3.3 Client

封装 Python 服务、对象存储、TTS、数字人等外部调用。

- 统一处理超时、重试、异常转换、日志。
- 不写业务决策逻辑。

### 3.4 Mapper

- 只负责数据访问。
- 不承载业务判断。
- 方法命名需表达业务语义。

## 4. 异常处理规范

- 统一使用全局异常处理器，如 `@RestControllerAdvice`。
- 自定义业务异常至少包含：错误码、错误信息、可选上下文。
- 参数错误、状态冲突、资源不存在、下游失败要区分异常类型。
- 不允许直接把 Exception message 原样返回前端。

**代码示例：全局异常处理**

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        log.warn("业务异常: code={}, msg={}", e.getCode(), e.getMessage());
        return Result.error(e.getCode(), e.getMessage());
    }
}
```

## 5. 日志规范

- **INFO**：主流程开始、结束、状态变更。
- **WARN**：可恢复异常、重试、参数异常趋势。
- **ERROR**：实际失败及堆栈。

**必打日志场景**：文件上传、解析任务创建、讲课会话开始/暂停/恢复/结束、问答调用、Python 服务调用。

日志中禁止打印密钥、完整 Token、用户敏感信息。

## 6. 不该做什么

- 不在 Controller 中写业务流程。
- 不让 Service 返回数据库 Entity 给前端。
- 不在 Mapper 中写业务规则。
- 不把外部 SDK 调用散落在多个业务类里。
- 不把状态值写成无注释的数字或字符串常量。
