# 01-项目总览与协作规约

本文档整合了项目背景、模块职责、通用命名规范及团队协作的基本原则，是全体成员的第一份必读规约。

## 1. 适用范围与基本原则

### 1.1 适用范围
- 本规约适用于以下目录：
  - `backend`
  - `python-service`
  - `frontend`
  - `docs`

### 1.2 基本原则
- **单一职责**：每一层只做该层应该做的事，不跨层堆逻辑。
- **先约定再实现**：接口、数据结构、状态流转先明确，再进入编码。
- **可观测性优先**：关键流程必须有日志、错误码、状态字段和链路标识。
- **面向协作编写**：代码要让别人能接手，不追求个人技巧式写法。
- **默认可测试**：新增逻辑应尽量设计成可以单测，不把核心逻辑塞进控制器或页面生命周期里。
- **兼容比赛交付**：文档、演示、部署方式与代码实现保持一致。

## 2. 仓库与模块职责

### 2.1 模块划分
- `backend`：Java 后端，负责统一网关、业务编排、状态管理、权限、数据持久化、对外 API 暴露。
- `python-service`：Python 智能服务，负责课件解析、向量检索、Agent 能力、算法型任务。
- `frontend`：Vue 前端，负责上传、讲课、问答、学习进度等交互页面。
- `docs`：需求、设计、接口、规约、规划等项目文档。

### 2.2 跨模块边界
- 前端不直接访问数据库，不绕过 Java 后端直连 Python 服务，除非明确设计为内部链路且已在文档中说明。
- Java 后端负责统一对外契约，Python 服务原则上作为内部能力服务。
- Python 服务不承担账号权限、平台级审计、前端页面状态编排等职责。
- 文档中的接口说明、字段名称、状态枚举必须与代码实现同步。

## 3. 通用编码规范

### 3.1 命名规范
- 文件、目录、接口路径、数据库表字段优先使用英文。
- 命名要求语义清晰，不使用 `temp`、`test1`、`data2` 这类无业务含义名称。
- 布尔变量使用可读前缀：`is`、`has`、`can`、`should`。
- 集合变量使用复数或集合语义命名，如 `coursewares`、`questionList`。
- 状态字段统一使用可枚举值，不允许用 `1/2/3` 这类无注释魔法值直接表达业务语义。

### 3.2 代码组织要求
- 单个函数只处理一个主动作，超过 50 行时应主动评估是否拆分。
- 超过 3 层嵌套时应重构，优先使用提前返回。
- 公共能力沉淀到公共模块，避免复制粘贴。
- 不允许保留大段注释掉的旧代码；历史请交给 Git。
- TODO 必须写清背景和后续动作，例如：`TODO: 接入流式回答后替换轮询逻辑`。

### 3.3 常量与配置
- 常量集中管理，不在业务代码中散落硬编码。
- 可变环境项必须走配置文件或环境变量，不写死在源码里。
- 第三方地址、Token、密钥、桶名、模型名、超时时间等统一配置化。

### 3.4 时间与时区
- 后端服务统一使用 ISO 8601 或标准时间戳表达时间。
- 存库默认使用统一时区策略，推荐 UTC 存储、展示层转换。
- 前后端联调时必须明确时间字段的格式、时区和单位（秒或毫秒）。

## 4. 敏捷协作流程

### 4.1 任务流转
1. 从 `docs/任务列表.md` 认领任务。
2. 基于 `develop` 分支创建 `feature/xxx` 分支。
3. 开发完成后，确保本地自测通过，发起 Merge Request。
4. **接口变更必做**：若涉及 API 字段增删改，**必须**同步更新 `docs/05-接口与API通信规约.md`。

### 4.2 每日站会关注点
- 是否有阻塞依赖？（如：前端等接口，算法等模型 Key）。
- 上下游接口字段是否对齐？
