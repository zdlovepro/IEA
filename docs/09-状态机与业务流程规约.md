# 09-状态机与业务流程规约

## 1. 需要状态管理的核心对象

- 课件
- 解析任务
- 讲课会话
- 问答会话
- 媒体生成任务（如 TTS 合成、视频切片）

## 2. 状态定义要求

- 每个状态机必须明确：
  - 状态枚举及可选值
  - 每个状态的含义
  - 允许迁移的路径（从哪到哪合法）
  - 失败后处理策略（回滚、重试、人工介入）
- 禁止在不同模块中对同一状态值赋予不同含义（例如 `SUCCESS` 在不同模块中不能既表示解析成功又表示支付成功）。
- 状态字段推荐使用字符串枚举，避免使用无注释的整数（如 `1`、`2`）直接暴露给前端或文档。

## 3. 核心状态机定义

### 3.1 课件状态（CoursewareStatus）

| 状态 | 含义 | 允许迁入 | 允许迁出 |
|------|------|----------|----------|
| `UPLOADED` | 已上传，待解析 | 初始状态 | `PARSING` |
| `PARSING` | 解析中 | `UPLOADED`、`PARSING`（重试） | `PARSED`、`FAILED` |
| `PARSED` | 解析完成 | `PARSING` | `GENERATING_SCRIPT` |
| `GENERATING_SCRIPT` | 讲稿生成中 | `PARSED` | `READY`、`FAILED` |
| `READY` | 可开始讲课 | `GENERATING_SCRIPT` | `UPLOADED`（重新解析） |
| `FAILED` | 处理失败 | `PARSING`、`GENERATING_SCRIPT` | `PARSING`（重试） |

### 3.2 讲课会话状态（LectureSessionStatus）

| 状态 | 含义 | 允许迁入 | 允许迁出 |
|------|------|----------|----------|
| `IDLE` | 未开始 | 初始状态 | `PLAYING`、`ENDED` |
| `PLAYING` | 讲解中 | `IDLE`、`RESUMING` | `INTERRUPTED`、`ANSWERING`、`ENDED` |
| `INTERRUPTED` | 被打断（如用户提问） | `PLAYING` | `ANSWERING`、`RESUMING` |
| `ANSWERING` | 问答中 | `PLAYING`、`INTERRUPTED` | `RESUMING`、`ENDED` |
| `RESUMING` | 恢复讲课中 | `INTERRUPTED`、`ANSWERING` | `PLAYING`、`FAILED` |
| `ENDED` | 已结束 | `IDLE`、`PLAYING`、`ANSWERING` | 终态，不可迁出 |

### 3.3 解析任务状态（ParseTaskStatus）

| 状态 | 含义 | 允许迁入 | 允许迁出 |
|------|------|----------|----------|
| `PENDING` | 待执行 | 初始状态 | `RUNNING` |
| `RUNNING` | 执行中 | `PENDING` | `SUCCESS`、`FAILED` |
| `SUCCESS` | 成功 | `RUNNING` | 终态 |
| `FAILED` | 失败 | `RUNNING` | 终态 |

## 4. 状态变更要求

- 每次状态变更必须在日志中记录：
  - 对象 ID（如 `courseware_id`、`session_id`）
  - 原状态
  - 新状态
  - 触发动作（如 `start_lecture`、`ask_question`）
  - 操作人/用户 ID
- 重要状态变更（如 `PLAYING` ↔ `INTERRUPTED`）必须落库保存。
- 前端展示状态必须来自后端权威结果，不得由前端自行推断或提前改变状态（例如倒计时结束自动认为讲课结束，而应等待后端状态）。

## 5. 状态流转代码实现建议

### 5.1 Java 后端（使用枚举 + 校验方法）

```java
public enum LectureSessionStatus {
    IDLE, PLAYING, INTERRUPTED, ANSWERING, RESUMING, ENDED;

    public boolean canTransitionTo(LectureSessionStatus target) {
        switch (this) {
            case IDLE: return target == PLAYING || target == ENDED;
            case PLAYING: return target == INTERRUPTED || target == ANSWERING || target == ENDED;
            case INTERRUPTED: return target == ANSWERING || target == RESUMING;
            case ANSWERING: return target == RESUMING || target == ENDED;
            case RESUMING: return target == PLAYING;
            default: return false;
        }
    }
}
```

在 Service 中：

```java
if (!currentStatus.canTransitionTo(newStatus)) {
    throw new BusinessException(ErrorCode.STATE_CONFLICT,
        String.format("不允许从 %s 切换到 %s", currentStatus, newStatus));
}
```

### 5.2 Python 服务（使用类常量 + 校验函数）

```python
class LectureSessionStatus:
    IDLE = "IDLE"
    PLAYING = "PLAYING"
    INTERRUPTED = "INTERRUPTED"
    ANSWERING = "ANSWERING"
    RESUMING = "RESUMING"
    ENDED = "ENDED"

    @classmethod
    def can_transition(cls, current, target):
        transitions = {
            cls.IDLE: {cls.PLAYING, cls.ENDED},
            cls.PLAYING: {cls.INTERRUPTED, cls.ANSWERING, cls.ENDED},
            cls.INTERRUPTED: {cls.ANSWERING, cls.RESUMING},
            cls.ANSWERING: {cls.RESUMING, cls.ENDED},
            cls.RESUMING: {cls.PLAYING},
        }
        return target in transitions.get(current, set())
```

## 6. 业务主链路状态流转示例

### 6.1 课件上传到可讲课

```text
UPLOADED → PARSING → PARSED → GENERATING_SCRIPT → READY
                ↓                             ↓
              FAILED                        FAILED
```

### 6.2 讲课中断与恢复

```text
PLAYING → INTERRUPTED → ANSWERING → RESUMING → PLAYING
              ↓            ↓
           (超时/失败)  (失败回退)
              ↓            ↓
            ENDED       INTERRUPTED (保留)
```

## 7. 状态冲突与异常处理

- 重复操作：如已处于 `PLAYING` 再次调用“开始讲课”应返回 `40901` 状态冲突。
- 非法迁移：如从 `ENDED` 尝试恢复，应返回业务异常并说明。
- 下游失败导致状态无法推进：保留原状态或转为 `FAILED` 并记录原因，不允许直接丢弃异常使状态停留在中间态。

## 8. 状态机测试要求

- 必须编写单元测试覆盖所有合法迁移路径。
- 必须测试非法迁移是否被正确拒绝。
- 必须测试状态变更落库和日志记录是否触发。
