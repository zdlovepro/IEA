
# 07-测试与质量保障规约

## 1. 测试层次与范围

### 1.1 Java 后端
- **单元测试**：Service 层核心逻辑、工具类、状态机流转、异常分支。
- **集成测试**：Controller 接口（含鉴权）、Mapper 数据访问、下游 Client Mock 调用。
- **组件测试**：关键业务流程串联（如上传→解析→开始讲课→问答）。

### 1.2 Python 服务
- **单元测试**：解析函数、Prompt 组装、结果校验与兜底逻辑。
- **接口测试**：FastAPI 路由的请求/响应结构、错误处理。
- **模型输出校验测试**：确保大模型返回经过强校验和格式清洗。

### 1.3 前端
- **单元测试**：Pinia Store 逻辑、工具函数、常量映射。
- **组件测试**：核心交互组件（上传、问答、会话控制）。
- **E2E 测试（可选）**：主链路（上传→讲课→问答）使用 Playwright/Cypress。

## 2. 测试工具与框架

| 模块 | 单元测试 | 集成/接口测试 | Mock |
|------|----------|----------------|------|
| Java 后端 | JUnit 5 + Mockito | SpringBootTest + TestRestTemplate | Mockito / MockBean |
| Python 服务 | pytest | pytest + httpx (AsyncClient) | unittest.mock / pytest-mock |
| 前端 | Vitest / Jest | Vue Test Utils | vi / jest |

## 3. 测试覆盖重点

### 3.1 必须覆盖的场景
- **上传与解析主链路**：文件上传 → 任务创建 → 解析完成 → 状态更新。
- **讲课会话状态机**：IDLE → PLAYING → INTERRUPTED → RESUMING → ENDED 每一条合法路径。
- **问答流程**：文本提问 → Python 调用 → 返回结构校验（answer + evidence）。
- **下游异常兜底**：
  - Python 服务超时 → Java 返回 502 错误，不崩溃。
  - Python 返回非 JSON 或缺失字段 → 使用默认兜底结果。
  - 对象存储下载失败 → 重试 + 降级提示。
- **并发与幂等**：重复开始讲课、重复解析同一课件应返回状态冲突。

### 3.2 非功能性测试关注点
- 大文件上传（>50MB）不超时。
- 讲课会话恢复后节点位置正确。
- 日志中无敏感信息（密钥、完整 token）。

## 4. 测试代码规范

### 4.1 命名与结构
- 测试类名：`XxxTest` 或 `XxxIT`（集成测试）。
- 测试方法名：`should_ExpectedBehavior_When_StateUnderTest` 或 `testMethodName_Scenario_ExpectedResult`。
- 使用 `@DisplayName` 提供中文说明（Java）。

### 4.2 示例：Java Service 单元测试

```java
@ExtendWith(MockitoExtension.class)
class LectureServiceTest {

    @Mock
    private CoursewareMapper coursewareMapper;
    @Mock
    private PythonServiceClient pythonClient;
    @InjectMocks
    private LectureServiceImpl lectureService;

    @Test
    @DisplayName("课件状态为 PARSING 时开始讲课应抛出业务异常")
    void should_ThrowBusinessException_When_CoursewareNotReady() {
        CoursewareEntity entity = new CoursewareEntity();
        entity.setStatus(CoursewareStatusEnum.PARSING.getCode());
        when(coursewareMapper.selectById("cware_123")).thenReturn(entity);

        StartLectureRequest request = new StartLectureRequest("cware_123", "user_001");
        BusinessException ex = assertThrows(BusinessException.class,
                () -> lectureService.startLectureSession(request));

        assertEquals(ErrorCode.COURSEWARE_NOT_READY, ex.getCode());
    }
}
```

### 4.3 示例：Python 接口测试

```python
import pytest
from httpx import AsyncClient
from app.main import app

@pytest.mark.asyncio
async def test_parse_endpoint_success(mocker):
    mocker.patch("app.services.parse_service.parse_ppt", return_value={"pages": 5})
    async with AsyncClient(app=app, base_url="http://test") as client:
        resp = await client.post("/python/v1/parse", json={
            "courseware_id": "test_001",
            "file_url": "s3://bucket/test.pptx"
        })
    assert resp.status_code == 200
    data = resp.json()
    assert data["code"] == 0
    assert "pages" in data["data"]
```

### 4.4 示例：前端 Store 测试 (Vitest)

```typescript
import { setActivePinia, createPinia } from 'pinia';

describe('lecture store', () => {
```

## 5. 测试数据与 Mock 约定

- 使用固定测试数据工厂（Factory）或 Builder 模式，避免在测试代码中重复构造复杂对象。
- 对外部依赖（数据库、Python 服务、对象存储）默认使用 Mock，除非是明确的集成测试环境。
- Mock 返回的响应结构必须与真实接口保持一致，不要简化字段。
- 时间相关断言允许 ±1s 误差，使用 `assertThat(duration).isBetween(100, 200)` 等形式。

## 6. 覆盖率要求（推荐）

| 模块 | 行覆盖率 | 分支覆盖率 | 说明 |
|------|----------|--------------|------|
| Java 核心 Service | ≥ 70% | ≥ 60% | 状态机、异常分支必须覆盖 |
| Java 工具类 | ≥ 80% | ≥ 70% | 纯函数逻辑 |
| Python 服务核心函数 | ≥ 70% | ≥ 60% | 解析、校验、兜底 |
| 前端 Store / utils | ≥ 60% | - | 核心逻辑 |

> 覆盖率作为参考指标，不强制 100%，但新增代码应尽量覆盖正常和异常分支。

## 7. 测试环境与数据隔离

- 单元测试使用 H2（Java）或 SQLite（Python，可选），不依赖外部数据库实例。
- 集成测试使用独立的测试数据库（如 test_*），每次运行前自动清理或回滚事务。
- 前端测试使用 Mock Server（MSW）或 API 拦截，不调用真实后端。

## 8. 持续集成（CI）要求

- 每次提交到 `develop` / `feature/*` 分支时，自动触发：
  - Java 后端单元测试 + 集成测试（关键模块）
  - Python 服务单元测试 + 接口测试
  - 前端 lint + 单元测试
- 合并到 `main` 前必须通过所有测试（CI 绿灯）。
- 不允许跳过测试强行合并。

## 9. 手动测试与验收

- 提交前必须完成以下场景的手动验证（若自动化测试未覆盖）：
  - 上传一个真实 PPT/PDF，等待解析完成。
  - 开始讲课，触发中断（语音提问），检查恢复后节点正确。
  - 连续问答 3 次，验证答案不重复且带证据。
- 在 PR 描述中附上手动测试截图或录屏。

## 10. 缺陷修复流程

- 发现 Bug → 先写一个**复现测试**（红色）。
- 修复代码 → 测试变绿（绿色）。
- 提交时附带测试用例，避免回归。
- 严重线上 Bug 需补充故障报告，记录根因和修复方案。

## 11. 禁止事项

- 不在测试代码中使用 `Thread.sleep()` 等待异步结果，应使用 `await`、`CountDownLatch` 或重试断言。
- 不忽略测试异常（如 `@Test(expected = ...)` 应改为 `assertThrows` 显式捕获）。
- 不因为测试“麻烦”而跳过对核心分支的覆盖。
- 不把集成测试和单元测试混在同一个测试类中而不做区分。
